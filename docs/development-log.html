产品定位

EZCopy 面向自媒体创作者，用于把 iPhone 拍摄的照片和视频通过本地 Wi-Fi 传到电脑。它不走云端，不依赖第三方服务器，不主动使用蜂窝数据，也不定位为影视工业 DIT 软件。

Wi-Fi LAN Chrome / Edge iPhone Photos ZIP Package No Cloud Transfer

阶段时间线

1

可行性与范围确认

明确产品只服务手机拍摄素材，不做影视工业级 DIT。确认接收端只支持 Chrome 和 Edge。

  • 确认局域网传输、禁止云端中转。
  • 确认不能保证所有公共 Wi-Fi 都开放设备互访。
  • 识别 Chrome 文件夹写入需要安全上下文。
2

项目基础搭建

创建 iOS SwiftUI 工程、浏览器接收端原型、共享 Manifest Schema 和文档目录。

  • 核心工程：ios/EZCopy.xcodeproj
  • 核心服务：TransferServer.swift
  • 选择逻辑：MediaSelectionViewModel.swift
3

相册选择与元数据

实现 PhotosPicker 选择照片和视频，并读取基础元数据。

  • 文件名、类型、创建时间、尺寸、视频时长。
  • 修复缺少 Photos asset identifier 导致的 500 错误。
  • 引导用户开启完整相册访问。
4

iPhone 局域网服务

通过 Network Framework 在 iPhone 上启动本地 HTTP 服务，电脑浏览器打开端口即可访问。

  • 默认端口：8080
  • Wi-Fi 地址检测：无 Wi-Fi 时不允许启动。
  • 已实现 /manifest.json、/download、/archive.zip。
5

从文件夹写入改为 ZIP 基础档

由于 http://iPhone-IP:8080 不是安全上下文，Chrome / Edge 无法稳定使用文件夹选择写入。最终选择旧端口模式 + ZIP 下载作为 MVP 基础档。

  • 保留旧端口体验。
  • 不使用本地接收页。
  • 浏览器通过普通下载保存传输包。
6

传输包与报告

实现无压缩 ZIP 传输包，包含素材、Manifest、报告和可选 MD5。

  • 目录：Photos/、Videos/
  • 文件：EZCopy_Manifest.json、EZCopy_Report.html
  • 可选：EZCopy_Checksums.md5
  • 支持 ZIP64，应对大视频文件。
7

缓存管理与可选 MD5

解决传输后 ZIP 和临时导出文件残留问题，并把 MD5 改成用户可选。

  • 下载完成或中断后自动清理本次 ZIP。
  • App 内加入 Clear Cache。
  • 加入 Generate MD5 checksums 开关。
  • 关闭 MD5 后跳过额外读取大文件的校验流程。
8

品牌与 UI 现代化

使用 Image 2.0 生成 App Icon 方向，选定复古人物 + EZ 字体方案，并重构 App 主界面。

  • App Icon 已加入 Assets.xcassets。
  • UI 采用 Utility Pro 结构，少量吸收 Retro Tech 色彩。
  • 修复浅底白字对比度问题。
  • 底部加入 EZCopy · Created by BLKero · 2026 隐约签名。

关键限制

  • 旧端口 HTTP 模式无法直接使用浏览器文件夹写入能力。
  • 当前 ZIP 方案需要手机临时空间，接近所选素材体积。
  • iCloud-only 原片不会自动联网下载，以避免非预期网络使用。
  • 公共 Wi-Fi、酒店 Wi-Fi、企业 Wi-Fi 可能阻止设备互访。
  • 当前 DIT 报告定位为自媒体传输记录，不是影视工业标准 DIT 报告。

下一步建议

  • 用多个大视频做真机压力测试。
  • 在生成 ZIP 前预估包大小和手机剩余空间。
  • 加入传输速度、预计剩余时间。
  • 优化浏览器端旧端口页面的展示和提示。
  • 研究流式 ZIP，减少手机临时空间占用。
  • 后续评估 HTTPS 本地服务，实现高级文件夹写入模式。
